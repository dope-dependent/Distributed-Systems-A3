Design of Raft Objects

Each partition inside a broker will serve as a Raft object.

Raft objects are implemented by inheriting them from the SyncObj class (with modifications) from the PySyncObj project. SyncObj handles all the steps of the Raft consensus protocol including leader election, request vote and append log. Every message sent from the Raft object is passed to the Raft Process (a.k.a the broker), along with the ID of the Raft object.

Every message that is arriving for a particular Raft object on the Raft process is passed to the onMessageReceived method of the SyncObj.

Periodic calls are made from the Raft process to the onTick method of each Raft object. This method is responsible for leader election.

Design of Raft Processes

Each broker is a Raft process.

Each Raft process has a TCP Server listening at the port __raftPort specified when the Raft process is initiated. Whenever a different Raft process tries connecting to the Raft process, a new object of the TCPConnection class is initiated which is stored in the Raft process, in its __connections dictionary.

The TCPConnection class is a part of the PySyncObj project and has methods to send, receive and parse messages. The Raft process uses these methods to communicate with other Raft processes. Every message that is received is passed to the __processConnection method in which the message is parsed and forwarded to the respective Raft object for which it is intended.

To create a new connection with another Raft process, a TCPConnection object is initiated and it is connected to the other Raft process.

Design of Partitions

The number of partitions for a topic is to be specified while creating the topic (defaulting to 1 if not specified). The partitions are assigned to the active brokers in a round robin manner. If a broker holding a partition is inactive, and a client tries to access it, we return an error message specifying that the partition is inactive. We do not migrate the partitions because it is assumed that the only time we will need to do it is when the broker hosting the partition is inactive in which case we have no means of accessing the partitions. Both producers and consumers can specify which partition they want to write into and read from respectively. However if nothing is specified, the message is produced randomly into any of the active partitions and is read from the partitions in a round robin manner. To read in a round robin manner, we store a lastindex variable for each consumer to remember the partition it last read from.  

Assignment of Partitions to Brokers

Since the replication factor is fixed at 3, three new active brokers with the least load are found. A replica of the new partition is created on each of the three active brokers selected.

Crash Fault Tolerance

We are using Raft for Crash Fault Tolerance. If f faults are possible, then 2f + 1 nodes are needed. Since our replication factor has been fixed at 3, we are supporting up to 1 crash fault for each partition. If two brokers having one partition in common are down, then none of the write operations on any of the replicas take place.

Creating and Removing Brokers

We believe that the administrator must have control over the creation and the removal of brokers and this should not be the responsibility of the users (producers and consumers) interacting with the broker manager. For this, docker-compose.yml needs to be changed.

Synchronizing Mechanism

We are considering the database corresponding to each entity as a state machine. Any operation changing the database (i.e. the state of the database) like INSERT, CREATE TABLE, and UPDATE will be called by a special function which is replicated across all the entities participating in the Raft consensus protocol.

Bonus

Q: Is the Broker Manager a single point of failure currently? If yes, how can we avoid that?

A: Yes, the Write Broker Manager is a single point of failure as upon its failure, we can neither send messages, nor create clients or topics. This can be avoided by having replicas of the Write Broker Manager or detecting the failure of the Write Broker Manager and promoting one of the Read Broker Manager to the status of Write Broker Manager.

Q: Would you use Raft for managers as well? If yes, would there be one Raft cluster for all read/write managers or separate clusters for read and write managers?

A: Yes, Raft would be used for managers. However, there would be no distinction between Read and Write Broker Managers. This is because all both Read and Write Managers are changing the state of the database and they need to be synchronized in order for correctness and consistency to be maintained in the system.

Q: Would the other read replicas detect the primary manager going down and take its role? Would the secondary have up-to-date information in this case?

A: Yes, since there is no distinction between Read and Write replicas, they are all participating in the Raft consensus protocol, which inherently detects crash fault in the leader and runs leader election protocol to promote one of the followers to the leader. However, the follower managers may not always have the most up to date information. This is possible in the following scenario:

Suppose the leader gets a message from the client which it will append to its log. However, before the AppendEntry message is sent to all the followers, the leader fails. In this case, the latest information is not propagated to all the followers and upon respawning of the leader, that log entry might get overwritten.

Q:  What other design choices can be made to improve the overall system's scalability and reliability while not giving up on the consistency?

A: Increasing the replication factor would increase the reliability of the system however it would slow down the system because of a lot of message passing overhead. The metadata can be redesigned in a way that the Write and Read Broker Managers are not editing the same data and hence can be put into different clusters.
